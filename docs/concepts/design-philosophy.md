# 설계 철학

데네브는 범용 프레임워크가 아니다. 단 한 명의 사용자를 위해, 단 하나의 채널에서, 단 하나의 하드웨어 위에 동작하도록 설계된 개인 AI 에이전트다. 이 문서는 모든 설계와 코드 결정의 근거가 되는 핵심 원칙을 정리한다.

## 단일 사용자 전제

데네브는 정확히 한 명의 운영자가 한 명의 사용자로서 사용한다. 멀티테넌트, 멀티유저, 팀 배포는 고려 대상이 아니다.

- 사용자 격리, 권한 분리, 동시 접근 제어 같은 코드 경로는 불필요하다
- 복잡한 동시성 처리보다 단순한 순차 처리를 우선한다
- 온보딩 플로우와 설정 마이그레이션을 최소화한다 — 운영자가 직접 설정한다

## 깊이 우선

넓게 얕게보다 좁게 깊이. 적은 것을 잘 지원하는 쪽을 항상 선택한다.

- **텔레그램**이 유일한 프로덕션 I/O 채널이다. 코드베이스에 다른 채널이 존재하지만 배포 대상은 아니다
- 새 플랫폼으로 확장하기보다 텔레그램 경험을 완벽하게 만드는 데 집중한다
- 텔레그램 Bot API 제약(4096자 메시지 한도, MarkdownV2 파싱, 인라인 키보드, 50MB 파일 제한)에 맞춰 최적화한다

## 의견 있는 기본값

Apple의 철학을 따른다: 프로그램 수준에서 설정을 고정하여 안정적이고 예측 가능한 사용자 경험을 제공한다.

- 좋은 기본값이 많은 옵션보다 낫다
- 사용자가 자신의 경험을 저하시킬 수 있는 설정 노브를 노출하지 않는다
- 설정을 추가하기 전에 자문한다: "이것이 정말 사용자에게 선택권을 줘야 하는가?"
- 견고함은 더 많은 옵션이 아니라 더 적은 움직이는 부품에서 나온다

## 높은 완성도와 응집도

출시하는 모든 기능은 완전히 완성되고 긴밀하게 통합된 상태여야 한다.

- 부분 구현보다 미구현이 낫다
- 기능 간 경계가 자연스럽고 빈틈이 없어야 한다
- "나중에 마무리하겠다"는 출시 기준이 아니다

## 하드웨어 최적화

데네브는 NVIDIA DGX Spark(Grace Blackwell) 위에서 동작한다. 모든 서비스가 이 단일 머신에서 실행된다.

- 로컬 GPU 추론을 적극 활용하고 외부 API 호출을 최소화한다
- 메모리와 GPU 리소스가 풍부하므로 공격적 캐싱과 프리로딩을 활용한다
- 빌드 타겟: `make gateway-dgx` (Vega FTS + 시맨틱 검색 + CUDA GGUF 추론)

## 바이브 코딩

모든 개발은 Claude Code와 AI 에이전트를 통해 이루어진다. 별도의 사람이 직접 작성하는 코드 워크플로우는 없다.

- 다음 AI 세션이 이어서 작업할 수 있도록 충분한 컨텍스트와 주석을 남긴다
- 복잡한 로직은 작고 명확한 이름의 함수로 분리하여 AI가 쉽게 이해하고 수정할 수 있게 한다

## 철학 부채

현재 코드베이스에서 위 원칙과 완전히 일치하지 않는 영역이 있다. 이는 "지금 당장 리팩토링하라"는 뜻이 아니라, 해당 영역을 다음에 건드릴 때 이 방향으로 단순화하라는 점진적 가이드라인이다.

- **설정 과다**: 게이트웨이와 텔레그램 채널의 설정 필드가 단일 사용자 전제에 비해 많다. 기본값을 고정하여 설정 표면을 축소하는 방향으로 진행한다
- **멀티채널 동등 표현**: 문서와 코드가 모든 채널을 동등하게 취급한다. 텔레그램을 명시적 tier 1으로 구분하는 방향으로 진행한다
- **범용 하드웨어 프로파일**: 여러 하드웨어 프로파일이 존재하나 프로덕션은 DGX Spark만 사용한다. DGX Spark을 기본으로 고정하되 폴백은 유지하는 방향으로 진행한다
- **인증 복잡도**: RBAC, rate limiting, trusted proxies 등이 단일 사용자에 비해 과도하다. 단일 사용자 모드에서 인증 경로를 단순화하는 방향으로 진행한다
