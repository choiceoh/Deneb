## 2. 아키텍처 평가

### 강점

**명확한 레이어 분리.** Protobuf → Rust core (FFI staticlib) → Go gateway → Telegram의 빌드/런타임 의존 방향이 일관적이고 역방향 의존이 없음. Rust가 알고리즘/검증, Go가 오케스트레이션/I/O를 담당하는 분업이 적절함.

**상태 머신 기반 FFI.** `*_new() → *_start() → *_step() → *_drop()` 패턴으로 Rust 내 모든 알고리즘 로직을 유지하면서 DB/LLM I/O는 Go 호스트에 위임. CGo 경계를 넘는 복잡도를 잘 관리함.

**Hub 배선 규칙.** `GatewayHub`는 서비스 컨테이너 역할만 하고 비즈니스 로직 없음. `method_registry.go` 단일 지점 배선, 핸들러 패키지의 Hub 미참조 등 규칙이 코드와 테스트로 강제됨.

**Lock-free RPC 디스패치.** `atomic.Value` 기반 핸들러 스냅샷으로 읽기 경로에 잠금 없음. 부팅 시에만 mutable staging 사용.

**듀얼 빌드 모드.** CGo FFI (`*_cgo.go`) + pure Go 폴백 (`*_noffi.go`)으로 Rust 빌드 없이도 Go 단독 개발/테스트 가능.

### 주의할 점

**실행 루프 복잡도.** `chat/run_exec.go` (1,292 LOC)에 모델 폴백, 컴팩션 재시도, 도구 디스패치가 모두 모여 있음. 단일 함수의 인지 부하가 높지만, 실행 흐름의 원자성 유지를 위한 의도적 선택으로 보임.

**대형 파일 분포.** Go 19개, Rust 14개 파일이 500 LOC 초과. 대부분 생성 코드/테스트 제외 시 실질적 로직 파일이며, 가이드라인(700 LOC)을 약간 초과하는 수준.

## 4. 특기할 엔지니어링 결정

### 극도로 절제된 의존성 (Go 직접 3개)
`go.sum` 59줄. protobuf, SQLite, WebSocket만 직접 의존. HTTP 라우팅도 `net/http` stdlib 직접 사용. 프레임워크 부재가 아닌 의도적 선택으로, 단일 사용자 게이트웨이에 적합한 운영 복잡도.

### SIMD 최적화 (aarch64 NEON)
DGX Spark의 Neoverse V2 코어에 맞춘 2x-unrolled NEON FMA로 코사인 유사도 계산. 스칼라 폴백으로 CI 호환성 유지.

### 릴리스 속도
5일간 10회 릴리스 (v4.2.2 → v4.9.0). release-please 자동화 + `make check` 게이트 + 12개 CI 워크플로우가 이 속도를 안전하게 지탱.

### 300개 데이터 드리븐 품질 테스트
단위 테스트를 넘어 실제 게이트웨이에서 한국어 응답 품질, Telegram 안전성, 도구 사용 완결성, 에지 케이스 안정성을 검증. "단위 테스트 통과 ≠ 제품 품질" 철학의 실현.

### 자율 최적화 루프 (Autoresearch)
LLM이 직접 상수/프롬프트를 반복 수정하고 metric을 측정하는 자동화된 최적화 인프라. Exploration → Exploitation 전략 전환, 연속 실패 시 자동 전략 변경.
