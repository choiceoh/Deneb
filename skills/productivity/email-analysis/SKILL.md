# 메일 분석

수신 메일 한 건을 업무 관점에서 심층 분석하는 절차서. 단순 요약이 아니라 **맥락·관계·리스크·행동**을 복원한다.

## When to Use

- 새 메일이 도착해 분석이 필요할 때
- 사용자가 특정 메일을 붙여넣고 "분석해줘"라고 할 때
- 진행 중인 프로젝트·거래 관련 메일을 검토할 때

쓰지 않는 경우: 단순 회신/발송, 일반 대화, 일일 브리핑(→ `morning-letter`).

## Procedure

### 1단계: 컨텍스트 수집

분석 전에 위키에서 관련 이력을 먼저 조회한다. 메일에 등장하는 **프로젝트명·회사명·인물명**으로 `wiki` 검색 → 과거 진행 상황, 인물 관계, 거래 조건을 확보한다. 이력이 있어야 "왜 지금"을 복원할 수 있다.

같은 스레드의 이전 메일도 반드시 확인한다. `gmail` 도구의 `thread` 액션(`message_id` 전달)으로 같은 대화 전체를 시간순으로 불러와 **이전 합의 → 변경 → 현재 요청**의 흐름을 추측이 아닌 근거로 파악한다.

### 2단계: 본문 요약

발신자가 무엇을 **요청 또는 통보**하는지 2~3문장. 첨부파일이 있으면 무엇인지 명시.

### 3단계: 타임라인 재구성 ("왜 지금 왔나")

이 메일이 **고립된 사건이 아님**을 전제한다. 위키 이력 + 메일 스레드 + 헤더 날짜로 관련 이벤트를 시간순으로 엮어 "왜 지금 이 메일이 왔는가"를 설명한다.
- 근거 있는 이벤트만 나열한다. 날짜·사실을 **추측으로 채우지 않는다**.
- 직전 액션과 이번 메일의 인과를 연결한다 (예: "수정 요청 → 후속 응답").

### 4단계: 이해관계자 매핑

누가 누구에게, 어떤 직책·역할로 보냈는지. 실무↔관리 라인, 의사결정권자, CC 수신자의 의미를 짚는다. 담당자 교체나 **에스컬레이션 신호**(실무자 위로 관리자가 직접 개입 등)가 보이면 명시한다.

### 5단계: 프로젝트 현황

이 메일이 속한 프로젝트가 **지금 어디에 있는지** — 진행 단계, 완료된 것, 대기 중인 것을 정리한다.

### 6단계: 리스크·기한 플래깅

결제 기한, 마감일, 금액, 미해결 이슈를 ⚠️로 눈에 띄게 표시한다. 시간에 민감한 항목(D-day 등)을 놓치지 않는다.

### 7단계: 다음 단계

추측이 아닌 **구체적 액션 1~3개**. 가능하면 담당자를 명시한다 ("이시연 — FTA CO 교체 확인").

### 8단계: 위키 갱신

분석으로 새로 확인한 사실(프로젝트 상태 변화, 인물 관계, 거래 조건)을 `wiki`에 갱신한다. 다음 분석의 연속성을 위한 필수 단계다.

- 금액·납기가 걸린 건별 거래는 `거래` 카테고리로 페이지를 만든다. 페이지 구조는 `templates/deal-page.md` 골격을 따른다.
- 결제기한·마감일이 있으면 `wiki write`의 `due` 필드(YYYY-MM-DD, 가장 임박한 날짜)에 기록한다 — 모닝레터의 "⏰ 임박 마감" 스캔이 이 값을 읽는다.
- 단가·수량·금액이 이전 메일과 달라졌으면 변경 전후 값을 모두 남긴다 ("단가 150원 → 145원, 5/21 변경").

## Output Format

한국어로, 아래 골격을 따른다. 해당 정보가 없는 섹션은 생략한다.

```
📬 {핵심 제목 한 줄}

발신: {이름/직책}
수신: {이름} (CC: {참조자})
시간: {수신 시각}

📝 본문 요약
{2-3문장}

🔍 왜 지금 왔나
{타임라인: 관련 과거 이벤트 → 이번 메일까지의 흐름}

📊 프로젝트 현황
{진행 단계, 완료/대기 항목}

👥 인물 관계
{핵심 인물, 직책·역할, 에스컬레이션 신호}

⚡ 다음 단계
1. {담당자} — {액션}
2. ...
⚠️ {기한·금액·리스크가 있으면 강조}
```

## Pitfalls

- **채널 상태를 추측하지 마라.** "텔레그램이 연결 안 되어서", "여기에 직접 전달", "채널 미연결" 같은 문구를 **절대 사용하지 마라** — 지금 응답하는 곳이 곧 사용자의 채널이다.
- **타임라인을 날조하지 마라.** 위키·스레드·헤더에 근거가 있는 이벤트만 나열한다. 모르면 "이력 확인 안 됨"으로 둔다.
- **AI 필러 금지.** "좋은 질문입니다", "물론이죠" 같은 서두 없이 분석부터 시작한다.
- **내부 토큰 누출 금지.** `<thinking>`, `NO_REPLY`, 함수 호출 마크업이 응답에 새지 않게 한다.
- **Telegram 4096자 제한.** 초과하면 본문 요약과 타임라인을 압축한다.
- 8단계 위키 갱신을 건너뛰지 마라 — 갱신이 없으면 다음 분석의 맥락이 끊긴다.

## Verification

- 출력에 타임라인("왜 지금 왔나")과 다음 단계가 모두 포함되었는가
- 기한·금액이 있으면 ⚠️로 표시되었는가
- 추측성 날짜·사실이 들어가지 않았는가
- 위키 갱신이 수행되었는가 — 거래 페이지에 결제기한·마감일이 `due`로 기록되었는가
