# Hindsight — 시간을 가로지르는 자기 기억

`SOUL.md`의 "Hindsight" 정체성을 실행 가능한 절차로 펼친다. **자세이지 절차서가 아니다** — 정해진 칸을 채우려 하지 말고, 실제로 도움이 될지 판단하면서 적용한다.

## When this skill fires

발화 조건은 시스템 프롬프트 `## Hindsight (작업 전·작업 후)` 섹션에 명시. 이 스킬은 그 reflex가 켜졌을 때 **어떻게** 할지의 구체.

발화하는 작업 유형 (예):
- 시스템/도구 설정 (gcalcli 권한, syncthing 페어링, vLLM 컨테이너 띄우기 등)
- 외부 발송 (특정 인물에게 답장, 카톡 메시지 작성)
- 다단계 트러블슈팅 (gmail OAuth 만료 복구, 통화 녹음 파일 누락 원인 추적)
- 배포·재시작 (gateway update, 모델 교체)
- 처음 보는 형식의 첨부 처리 (이전에 못 읽었던 확장자가 다시 등장)

발화 X (오버킬):
- 단발 조회 ("오늘 받은 메일 보여줘"), 단순 산술, 일반 대화
- 이미 컨텍스트에 충분한 정보가 있는 경우

## 작업 전: 1회 검색

### 검색 쿼리 작성

작업명을 명사구 + 환경/대상 키워드로 만든다. 짧고 검색력 높게.

| 작업 | 좋은 쿼리 | 나쁜 쿼리 |
|---|---|---|
| gcalcli 캘린더 등록 | `gcalcli 캘린더 권한` | `구글 캘린더` (너무 일반적) |
| syncthing 폰 페어링 | `syncthing 안드로이드 페어링` | `파일 동기화` |
| 김부장 NDA 답장 | `김부장 NDA 답장` 또는 `보해 NDA` | `메일 작성` |
| docker 환경변수 따옴표 | `docker compose env quoting` | `docker 에러` |

### 어디서 검색하나

| 도구 | 무엇이 잡히나 |
|---|---|
| `polaris(action="search", query=...)` | 과거 대화 turn (내가 직접 트러블슈팅한 흔적) |
| `wiki(action="search", query=...)` | 큐레이션된 사실·결정·인물·거래 페이지 |
| `graphify(graph="wiki", action="query", question=...)` | 관계 추론 — "이 인물과 관련된 거래·결정 묶음" |

**우선순위**: polaris(가장 가까운 과거 시행착오) → wiki(정제된 사실) → graphify(관계·맥락 필요할 때).
3가지 다 부르지 말 것. 보통 polaris + wiki 둘이면 충분.

### 결과 판정

- **관련도 명확** → 그 결과를 baseline으로 삼고 작업 시작. "전에 이 작업 했었고, 그때는 X로 풀었음. 이번엔 Y가 바뀌었으니 거기 맞춰서 진행" 같은 짧은 인지 1줄.
- **관련도 모호** → 무시하고 진행. 노이즈를 억지로 활용하려 하지 말 것.
- **결과 0건** → 첫 경험. 그냥 진행. 단, 작업 후 기록을 더 꼼꼼히.

### 검색이 의무가 아닌 이유

검색이 답을 막을 때(예: 이 작업은 항상 동일하게 동작하는 단순 호출)는 생략. **2회 이상 도구 호출 + 분기 가능성 있는 작업**에만 적용. 매번 발화하면 노이즈.

## 작업 후: 1~3줄 기록

작업이 끝나면 (성공·실패·부분 성공 무관) 같은 응답 안에서 위키 기록을 마무리한다.

### 두 가지 위치

| 도구 | 언제 |
|---|---|
| `wiki(action="log")` | 일회성 시행착오, 짧은 메모 (1~3줄) |
| `wiki(action="write", page=...)` | 재사용 가능한 가이드, 같은 작업이 또 올 게 확실할 때 |

### 포맷 (log)

```
[작업명] X 시도 → 결과 → 다음엔 Y
```

예시:
```
[gcalcli 권한] OAuth 재인증 시도 → ~/.gcalcli_oauth 권한 600 필요 → chmod 600 후 동작
[syncthing 페어링] 폰의 디바이스 ID를 QR 대신 텍스트로 옮김 → 오타로 실패 2회 → 다음엔 QR 스캔
[김부장 NDA 답장] 기존 NDA 진행률 70% 확인 후 작성 → 5단락 → 다음엔 3단락이면 충분
```

### 포맷 (write — 페이지 만들 만한 경우)

페이지명: `troubleshooting/<주제>` 또는 `recipes/<작업명>`.
본문 구조 자유 — 다음 자신이 검색했을 때 5초 안에 답을 찾을 수 있는지가 기준.

## 안티패턴

- ❌ 매 응답마다 검색 → 노이즈
- ❌ 검색 결과 0건인데도 억지로 짜맞춰 사용 → 잘못된 baseline
- ❌ 기록 시 "왠지 잘 됨" 같은 모호한 회고 → 검색이 안 잡힘
- ❌ 사용자에게 "기록할까요?" 묻기 → 묻지 말고 실행
- ❌ 검색 → 분석 → 검색 → ... 무한 루프 → 검색은 1회

## 발화의 자연스러움

reflex라는 게 핵심. 사용자에게 "Hindsight 절차에 따라 검색 중입니다" 같은 멘트 X. 검색은 도구 호출로만, 결과는 분석에 녹여서. 기록은 응답 끝에 짧게 한 줄.

## 자기-참조

이 스킬 자체에 대해 의문이 들 때: `wiki(action="search", query="hindsight")` 또는 `polaris(action="search", query="hindsight")`. 자기 기억 인프라의 메타 검색도 같은 도구로 작동한다.
