# MOSS (arXiv:2605.22794v2) - Deneb 적용 분석

Source: <https://arxiv.org/html/2605.22794v2>
Read date: 2026-06-18
Doctrine mapping: `genesis.PropusDoctrine()` records this as the source-level
self-improvement coding rule for Propus.

## One-line take

MOSS is the clean boundary line Deneb needed: self-improvement that edits code
is not a skill, a prompt, or a lifecycle log row. It is a source-level candidate
that needs its own queue, review state, verification proof, consent gate, and
rollback path.

## Paper mechanics

MOSS argues that deployed agent systems cannot repair all failures by changing
text-mutable artifacts. Bugs in routing, hook ordering, session state,
dispatch, and promotion logic live in code, so they need source-level changes.

The system design has five relevant pieces:

1. A production agent substrate keeps serving users.
2. A control CLI exposes status, batch, start, stop, apply, flag, and catch-up.
3. A pluggable external coding-agent CLI performs the actual source edit.
4. A host daemon supervises builds, trial workers, and container swaps.
5. Ephemeral trial workers replay the evidence batch against the candidate.

The evolution loop is directed rather than exploratory. It starts from a concrete
batch of production-failure evidence, runs locate/plan/review/implement/review/
evaluate/verdict stages, and promotes only after verification. Promotion is
user-consent gated and guarded by a health-probe rollback path.

The paper's OpenClaw case study reports one iteration improving a four-task mean
grader score from 0.2526 to 0.6100 after a harness-level code patch. The useful
lesson for Deneb is not the number itself; it is that the winning fix landed in
the harness path where skill/prompt edits could not reach.

## What transfers to Deneb now

Directly transferable:

- **Separate product surface.** Source-level candidates should not be rendered
  as skills or ordinary Propus lifecycle events. They are a coding backlog.
- **Evidence-first rows.** Each candidate must preserve evidence, target files,
  proposed change, and risk, so an AI coding agent can review the queue without
  reconstructing context from chat history.
- **Status auditability.** Proposed, accepted, applied, rejected, and superseded
  states should stay visible. Hiding non-pending items loses the review trail.
- **External coding-agent boundary.** Deneb's runtime records candidates; a
  coding-specialized agent/model performs edits only after review and tests.
- **Promotion discipline.** Applied status should mean the patch passed review,
  tests, user/operator consent, and has an operational rollback path.

Not directly transferable yet:

- **Full trial-worker replay.** Deneb has local tests, render previews, CI, and
  production listener proof, but not yet isolated per-candidate trial workers.
- **Automated container swap.** Deneb still uses the existing PR/merge/deploy
  lane; that is safer than introducing host-daemon swaps prematurely.
- **Full MOSS CLI.** The current user need is native visibility and coding-agent
  review, not a separate `moss evo` command surface.

## Implemented in this pass

- `miniapp.self_improvement_coding.list` now returns status counts across the
  latest queue window and rejects unknown status filters instead of silently
  widening them.
- Settings `자가개선 코딩` now renders a dedicated status filter row for pending,
  accepted, applied, rejected, superseded, and all source-level candidates.
- `PropusDoctrine()` now includes MOSS as a canonical source and records the
  invariant that source-level self-corrections are not skill lifecycle events.

## Operating rule

For self-improvement coding, the default question is:

> Which production evidence batch justifies a source-level change, which files
> would change, what risk does it carry, and which review/test/rollback evidence
> allows promotion?
